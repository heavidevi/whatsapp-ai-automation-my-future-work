Pixie Humanization — Detailed Execution Plan

  Ground rules for this run

  1. One task at a time. Finish → test with a dumb user → only move on if the improvement lands.
  2. Dumb-user tests drive acceptance. Every task has a concrete scenario the test must pass. If it fails, fix and retest.
  3. Extend test-harness.js, don't rewrite it. Add new dumb-user scenarios per task.
  4. Everything stubs the Netlify deployer + sender. No real WhatsApp / real deploys during testing.
  5. Cleanup between runs. Test phone is randomized so no DB collision.
  6. No breaking changes to the completed Phase 1 work (messageAnalyzer, router intent interceptors, intentHelpers). We build on top.

  Task 1 — Objection handler (Phase 2.3 from plan.txt)

  New dedicated handler for intent === 'objection'. Validates concern → RAG-backed social proof → low-commitment next step. Track objection topics in metadata so the follow-up scheduler can
  pick it up.

  - New file: src/conversation/handlers/objectionHandler.js
  - Edit: src/conversation/router.js — when analysis.intent === 'objection' and user is in SALES_CHAT/collection, route through the handler instead of falling through.
  - Edit: src/db/users.js metadata schema — add objectionTopics: string[].

  Dumb-user test scenarios:
  1. Mid-webdev-flow: "this is too expensive, I'll just use wix" → bot acknowledges, shares value context, offers to continue or schedule call. Does NOT push for the sale.
  2. "not sure this is worth it" → empathetic validation, social proof, lower-commitment option.
  3. "let me think about it" → respects it, seeds follow-up (no fake urgency).

  Acceptance: transcript shows empathetic validation phrases ("I hear you", "totally fair"), social proof inclusion, zero pushy re-asks.

  Task 2 — Language detection + response switching (Phase 2.4)

  messageAnalyzer already returns language. Plumb it:
  - Store preferredLanguage in user metadata on first non-English message.
  - Pass it into every LLM prompt as a directive ("respond in Urdu").
  - If analyzer returns an RTL/non-Latin script we don't support well, stay in English with a polite acknowledgment.
  - Edit: src/conversation/router.js — after analyzer, persist language if confident (≥ 2 consecutive turns in same language).
  - Edit: src/llm/prompts.js — append language directive to sales/general/revision prompts.
  - Edit: src/db/users.js — metadata preferredLanguage.

  Dumb-user test: simulate an Urdu user ("Mujhe apne salon ka website chahiye"), bot should reply in Urdu for the rest of the session.

  Acceptance: last 5 bot messages are in user's language; preference persists in DB.

  Task 3 — Smart defaults for "skip" / "whatever" (Phase 5.3)

  When isSkip(message) or analysis.isDelegation is true at an optional collection step, apply a sensible industry default + tell the user what we picked + offer to change.

  - New file: src/config/smartDefaults.js — map industry → {colors, style, ...}
  - Edit: webDev.js handleCollectColors, adGeneration.js handleCollectColors, logoGeneration.js handleCollectColors, SALON_INSTAGRAM, etc.

  Dumb-user test:
  - Salon flow, hours step: "whatever you think" → bot applies Tue-Sat 9-7 default + states it + allows change.
  - Ad flow, colors step: "idk you pick" → bot picks pastel-bakery / tech-blue based on industry + confirms.

  Acceptance: bot never stalls on "whatever"; always states the chosen default.

  Task 4 — Entity accumulator module (Phase 3.1 + 3.2 done cleanly)

  Currently entity extraction is split: router sets extracted*, webDev hydrates, ad/logo/chatbot don't use it. Consolidate into one module.

  - New file: src/conversation/entityAccumulator.js
    - hydrateMetadata(user, analysis) — persists extracted entities (existing router inline code → moves here).
    - fillFromMetadata(flow, user) — returns a seeded data object for a given flow (webdev, adgen, logo, chatbot).
    - nextMissingField(flow, data) — same idea as nextMissingWebDevState but per flow.
  - Edit: router.js uses hydrateMetadata.
  - Edit: webDev.js uses the new helpers (replaces scattered auto-prefill blocks).
  - Test: backward-compat — existing webdev dumb-user test still passes.

  Dumb-user test: parameter-chain dump "My business is Fresh Cuts, we're a barbershop in Karachi, call me at 0300-1234567" in a single message → webdev jumps straight to services step
  (business name, industry, city, phone all captured).

  Acceptance: flow skips collection steps the user already answered, even if out-of-order.

  Testing protocol (applied to every task)                    
                                                                                                                                                                                              
  1. Run the existing dumb-user salon scenario — ensure nothing regressed.
  2. Run the task-specific scenario added to test-harness.js.                                                                                                                                 
  3. If both pass → commit + move on.                                                                                                                                                         
  4. If either fails → diagnose from the transcript, fix, retest. Max 2 fix iterations before flagging for review.
